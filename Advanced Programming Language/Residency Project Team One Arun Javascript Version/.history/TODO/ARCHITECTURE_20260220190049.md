# TODO App - Multi-User Architecture & Data Flow Documentation

## 🎯 Project Overview

**Advanced Programming Language - Residency Project**

- Multi-user TODO management application with task categorization
- Each user can create, manage, and assign tasks to other users
- Tasks are categorized and tracked with priority levels
- Full CRUD operations with persistent storage

## 👥 Multi-User Data Model

### Users Collection
```javascript
{
  "id": 1,
  "name": "Arun",
  "email": "arun@example.com",
  "color": "#667eea"  // User's color theme
}
```

### Categories Collection
```javascript
{
  "id": 1,
  "name": "Work",
  "color": "#667eea"  // Category color for visual coding
}
```

### Todos Collection (Enhanced)
```javascript
{
  "id": 1,
  "text": "Complete project documentation",
  "completed": false,
  "createdBy": 1,      // Which user created this task
  "assignedTo": 2,     // Which user this task is assigned to
  "categoryId": 1,     // Task category
  "dueDate": "2026-02-28",
  "priority": "high"   // Priority: high, medium, low
}
```

### Key Features:
- **User Context:** Each user sees only their assigned tasks
- **Task Assignment:** Users can create tasks and assign them to others
- **Priority Levels:** Tasks have priority indicators (🔴 High, 🟡 Medium, 🟢 Low)
- **Categorization:** Tasks are organized by predefined categories
- **Color Coding:** Categories and users have color associations

## 🔄 User-Centric Data Flow

### When User Switches (User Selector Changes)

```
USER SELECTS DIFFERENT USER
        ↓
setCurrentUser(userId)
        ↓
Update this.currentUserId
        ↓
Filter todos where assignedTo === currentUserId
        ↓
updateComponents()
        ↓
Re-render ToDoList with new user's tasks
        ↓
UI Updated - Show only assigned-to-user tasks
```

### When Task is Created (Multi-User Form)

```
USER FILLS FORM:
  - Title: "Design new feature"
  - Assign To: "Tarak" (userId: 2)
  - Category: "Work" (categoryId: 1)
  - Priority: "High"
        ↓
User clicks "Add Task"
        ↓
EVENT HANDLER (app.js)
  1. Collect form data:
     {
       text: "Design new feature",
       assignedTo: 2,
       categoryId: 1,
       priority: "high",
       createdBy: 1 (current user)
     }
  2. Call todoContainer.addTodoItem(todoData)
        ↓
BUSINESS LOGIC (ToDoContainer)
  1. Call todoService.addTodo(todoData)
        ↓
SERVICE LAYER (TodoService)
  1. POST /todos with full data object
  2. Include all fields: text, createdBy, assignedTo, categoryId, priority, dueDate
        ↓
JSON SERVER
  1. Generate ID
  2. Store in db.json
  3. Return created todo
        ↓
RESPONSE FLOW
  ↓ Update local todos array
  ↓ Filter for current user
  ↓ Re-render ToDoList
  ↓ Display "Task added and assigned to Tarak"
```

## 🔗 Data Flow for Each Operation

### 1. **FETCH/GET - Load All Todos**

```
User visits app → Browser loads index.html
                    ↓
                  app.js
                    ↓
          initializeApp() [async]
                    ↓
        todoContainer.initialize()
                    ↓
        todoService.getTodos()
                    ↓
        fetch("http://localhost:3000/todos")
                    ↓
        JSON Server: Read db.json
                    ↓
        Return todos array
                    ↓
        Store in this.todos
                    ↓
        updateComponents() → render UI
```

### 2. **CREATE/POST - Add New Todo**

```
User types text + clicks "Add"
                    ↓
        Event listener triggers
                    ↓
        todoContainer.addTodoItem(text)
                    ↓
        todoService.addTodo(text)
                    ↓
        fetch(POST, {text, completed: false})
                    ↓
        JSON Server: Generate ID, add to db.json
                    ↓
        Return new todo object with ID
                    ↓
        Push to this.todos array
                    ↓
        updateComponents() → re-render
```

### 3. **UPDATE/PATCH - Edit or Toggle Todo**

```
User clicks Edit/Checkbox
                    ↓
        Event listener triggers
                    ↓
        todoContainer.editTodoItem(id, text)
        OR
        todoContainer.toggleTodoItem(id)
                    ↓
        todoService.updateTodo(id, {updates})
                    ↓
        fetch(PATCH, {text or completed})
                    ↓
        JSON Server: Update db.json entry
                    ↓
        Return updated todo object
                    ↓
        Update local this.todos array
                    ↓
        updateComponents() → re-render
```

### 4. **DELETE/DELETE - Remove Todo**

```
User clicks Delete button
                    ↓
        Event listener triggers
                    ↓
        todoContainer.deleteTodoItem(id)
                    ↓
        todoService.deleteTodo(id)
                    ↓
        fetch(DELETE, /todos/{id})
                    ↓
        JSON Server: Remove from db.json
                    ↓
        Return success
                    ↓
        Filter out from this.todos array
                    ↓
        updateComponents() → re-render
```

## 🏗️ Architecture Patterns Used

1. **MVC Pattern** - Separation of concerns
   - Model: TodoService (data layer)
   - View: Components (Header, ToDoList, AddTodo, Stats, Footer)
   - Controller: ToDoContainer (business logic orchestrator)

2. **Service Layer Pattern** - TodoService abstracts all API calls

3. **Component-Based Architecture** - Modular, reusable components with render() methods

4. **Async/Await** - Clean asynchronous operation handling

5. **REST API** - Standard HTTP methods for CRUD operations

6. **Event Delegation** - Central event handling in app.js with re-attachment after DOM updates
